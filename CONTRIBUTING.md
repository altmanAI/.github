# Contributing to AltmanAI

Thank you for your interest in contributing to the AltmanAI ecosystem. This document defines how contributors, collaborators, and AI development assistants should propose changes across AltmanAI repositories.

## Before you start

- Read this repository's `GOVERNANCE.md` for the full decision process and approval requirements.
- Review `mirror.txt` in this repository if you are contributing to canonical documentation — downstream artifacts should not expand or reinterpret upstream decisions without approval.
- Public artifacts that reference the P.A.I.H.I. framework or Proof-of-Impact Ledger should be evidence-grounded, not aspirational.

## How to open an issue

Use the appropriate issue template:

- **Bug report** — something isn't working as expected
- **Feature request** — a new capability or improvement
- **Research proposal** — a research question or exploratory concept
- **Documentation update** — a correction or improvement to existing docs

Be specific. Include context, expected behavior, and — where relevant — screenshots, logs, or reproduction steps. Never include secrets, credentials, or private data.

## How to submit a pull request

1. Fork the repository and create a feature branch.
2. Make your change, following the documentation and quality standards below.
3. Reference any related issues in your PR description.
4. Fill out the pull request template completely, including the AI-assisted-work disclosure if applicable.
5. Open the PR. At least one maintainer review and passing checks are required before merge.

## Documentation quality standards

- Write in clear, professional Markdown.
- Avoid unverified claims, unsupported partnerships, or exaggerated status (e.g., don't describe something as "launched" or "certified" unless that status is confirmed in the repository).
- Conventional commit messages are preferred (`feat:`, `fix:`, `chore:`, `docs:`, `ci:`).
- Signed commits are recommended.

## Human review requirement

Every change — regardless of whether it was drafted with AI assistance — requires human review before merge. AI development assistants (including the AltmanAI SuperAgent 1.0 AI Technology Partner) may prepare branches, commits, and PR drafts, but may not merge into `main` without explicit approval from Blake Hunter Altman.

## AI-generated contribution disclosure

If a contribution was drafted, generated, or substantially assisted by an AI tool, disclose this in the pull request description under "AI-assisted work." This keeps the review process transparent and honest.

## No secrets, no private data

Never include API keys, tokens, passwords, personal contact information, or financial records in issues, pull requests, commits, or documentation. If you discover exposed secrets, report them per `SECURITY.md` instead of filing a public issue.

## Founder approval gate

All governance-level, public profile, or organization-standard changes require explicit sign-off from Blake Hunter Altman, confirmed with the phrase **"All Clear for Impact."** Product-level repositories may define lighter-weight review processes for routine changes.
